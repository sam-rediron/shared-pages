Section 01

The Actuals vs. Forecast Problem

This is the single biggest product issue heading into the pilot. It touches nearly every financial metric on the dashboard and has cascading implications for overrides, data trust, and the entire DOCF workflow.

How the data works today

ImpactOS pulls financial data from Sage Intacct, which contains only actuals — revenue and expenses posted to the general ledger. This is correct accounting data.

Separately, DOCFs maintain a forecast file (spreadsheet) outside of Sage containing working budgets, projected revenue, and forecasted expenses. This file is the basis for what DOCFs present to Executive Directors in weekly and monthly meetings.

There is also a distinction between the working budget (the DOCF's forecast file) and the approved budget (filed with the authorizer). What's historically been shown to clients is typically the approved budget side-by-side with actuals, but DOCFs use the working budget/forecast for planning and client conversations.

Why this creates a problem

Revenue recognition in Sage is lumpy. Revenue is only recognized when it hits the GL, which can lag significantly. State apportionment revenue may not be fully posted until 20+ days into the following month. At any given point during a month, Sage actuals for revenue-dependent metrics will understate what the DOCF considers the "real" number.

Real-world friction — Marisol (Allegiance DOCF)

During sign-off, Marisol overrode nearly every metric. Some were trivial (enrollment off by 1), but others were material:

  • Days Cash on Hand — System showed ~54 days; she wanted ~70. Her calculation includes projected/forecast revenue. Adam's response: "If you include projected revenue, it's a cash projection, not days cash on hand."
  • Revenue Per Student — System showed ~$8,000; she wanted ~$14,000. The gap is entirely due to revenue earned but not yet recognized in Sage.
  • Salaries as % of Revenue — She was fine with the actuals-based number. Even within a single DOCF, the rules about actuals vs. forecast are not consistent across metrics.

Jim (Peak Prep DOCF) had a different tolerance — comfortable with metrics being off by one or two, didn't require forecast-based overrides at the same scale.

What this means for the product

The override feature was designed for data correction — situations where the system calculates incorrectly due to non-standard GL coding, unusual account structures, or known data issues. But overrides are now being used for a completely different purpose: substituting forecast numbers for actuals because the DOCF wants the metric to match their forecast file. This conflation of two use cases in a single feature is a design problem.

Current state of the codebase

The system already has the architecture for this. metricValues.ts supports a valueType field (actual | forecast) and a view mode (actuals | forecast | actuals-plus-forecast). The blended mode uses a cutoff date (last closed accounting period) to show actuals for closed periods and forecast for open/future periods. PRD-03 specifies an Actuals/Forecast/Blended toggle on the dashboard.

Key insight

The gap is not architecture — it's data. Forecast data does not exist in the system for pilot schools. The toggle exists in the PRD and the filtering logic exists in code, but there's nothing to filter because nobody has uploaded forecast values. The override became the stopgap.

Minimum for pilot launch

  • A way to get forecast data into the system (upload/import or manual entry per metric)
  • Dashboard toggle or side-by-side view to show actuals alongside forecast
  • Clear labeling on each metric indicating which data mode is active
  • Methodology notes on revenue-sensitive metrics explaining what the number represents
Design tension

DOCFs prefer actuals and forecast side by side, not as a toggle. The concern about toggles is they hide information. The concern about side-by-side is information density on an already data-heavy dashboard. Both approaches are feasible.

Operational prerequisite: Adam committed to reaching out to David (COO) to get definitive clarity on how DOCFs should operate — which metrics use which data source, and what the "CI Way" standard is. Without this decision, the product team cannot finalize the design.

Broader context

This is not a new problem. The tension between forecast files and Sage actuals has existed at Charter Impact for five years. ImpactOS is now acting as the forcing function — the pain of maintaining dual systems is finally real enough that DOCFs are willing to revisit their operational workflows.

Section 02

Override Feature: Scope and Future

Current design

DOCFs can override any scalar metric's value for a school. The override replaces the displayed value, shows attribution ("Overridden by DOCF jyerkie on 5/14/2026"), and is visible to EDs. Overrides are entered with a rationale and can be reviewed/expired.

Two distinct use cases are conflated

1. Data Correction

The intended use case

The automated value is wrong due to non-standard GL coding, account mapping issues, or known data pipeline bugs. The DOCF is fixing a calculation error.

Should remain as override with amber indicator, rationale, and audit trail.

2. Forecast Substitution

The unintended use case

The automated value is technically correct (accurate Sage actuals) but doesn't match what the DOCF presents from their forecast file. The DOCF is replacing actuals with forecast.

Should be handled through actuals/forecast view mode — not overrides.

Override reporting and governance

GitHub issue #167 covers Metric Override Reporting — an admin-level view showing how many metrics are overridden, how frequently, and by whom. The strategic intent: use this data to identify chronically overridden metrics, investigate root causes, and gradually remove the ability to override specific metrics as underlying data or calculations are fixed.

The vision is progressive tightening: identify patterns, fix root causes (GL standardization, forecast import, methodology), then disable overrides for metrics that are reliably accurate. This prevents overrides from becoming a permanent crutch.

Precision tolerance

Jim considers "off by one" close enough. Marisol required exact matches to her forecast numbers, even when the difference was definitional (actuals vs. forecast), not a calculation error.

Adam's framing: if enrollment shows 957 and the DOCF's spreadsheet shows 958, that's a data investigation. That's different from a DOCF wanting to change Days Cash on Hand by 15 days because they're including projected revenue.

Section 04

Restricted Grants / Net Asset Spend-Down

What DOCFs and EDs want

Jim described a concrete, recurring workflow: roughly every two months, school leaders email asking how much they've spent on restricted grants and how much they have left. Jim manually compiles this. A KPI showing restricted grant spend would eliminate this entirely.

Jim's conviction: Josh at Peak Prep would "live on this KPI every day." Dr. Fernandez would check it "pretty often."

Why it was dropped from the initial metric set

The team previously had restricted grant spend-down data in the Domo instance (the legacy BI tool). The specific blocker: they had the spend side (how much has been spent per restricted grant) but not the balance side (how much is left). Katie pulled the balance for one location but described it as "onerous." Without both sides, the metric was held in the backlog.

Path forward

Unresolved question

Is the team okay with shipping the spend-only half of this metric (how much has been spent per restricted grant, from Sage actuals), knowing the balance/remaining side isn't available? This answers "how much have we spent?" even if not "how much is left?" — but depends on capacity and prioritization.

Section 06

The Budget Season Timing Problem

June is board budget season. Most EDs and school leaders will be "up to their eyeballs in budgets and presentations before boards." This creates a tension:

  • The pilot wants active client engagement with ImpactOS during June
  • Clients are maximally busy with their most important annual deliverable
  • However, budget season is also when metric visibility is most valuable — ADA trends, salary costs, restricted fund positions, revenue projections are exactly what's needed for budget decisions

The opportunity

As Adam framed it: use the pilot to discover whether ImpactOS is "modestly useful" or "highly useful" during budgeting. Key inputs for budgeting that could come from ImpactOS:

  • ADA — current year vs. prior year vs. two years ago (trend for enrollment/revenue projections)
  • Salary levels & student/teacher ratio over time
  • Unduplicated Pupil % trajectory (affects LCFF supplemental/concentration grant funding)
  • Restricted grant spend-down position (directly informs budget allocation decisions)

If ImpactOS can answer "what does ADA look like over the last three years?" and "how has student/teacher ratio shifted?" during a budget planning conversation, it becomes a tool the ED can't ignore. But that requires historical data in the system, which is partially available.

Section 07

The Legacy View and What It Provided

The older green dashboard (Domo-era) provided a fundamentally different presentation than the current ImpactOS card-based layout:

  • Revenue tables with category, title, GL/order code, and both actuals and budgeted side by side
  • Revenue Previous Month and Revenue YTD as separate sections
  • Expenses Previous Month and YTD Expenses in the same format
  • Cash Book Amount and Days Cash on Hand as standalone widgets
  • Location selector at the top

Legacy (Domo) — Tabular, Variance-Oriented

What DOCFs are accustomed to

Answers "where is my money going and how does it compare to plan?" — the fundamental question of school financial management. Shows actuals and budget naturally in table columns.

ImpactOS — Card-Based KPIs

The current design

Answers "what is the current number and is it healthy?" — more executive-summary and scannable, but doesn't yet provide the actuals vs. budget comparison DOCFs rely on.

This connects directly to the actuals/forecast problem: the legacy view showed both numbers naturally because it was a table. The card view has to make a design choice about whether to show one number with a toggle or two numbers side by side, and neither option is as clean as a table row with columns.

Section 08

ImpactOS as a Service Delivery Platform

A strong theme emerged around ImpactOS being more than a dashboard — it should be the vehicle for standardizing how Charter Impact delivers service.

The "CI Way" opportunity

Charter Impact has not systemized:

  • What a standard weekly meeting agenda should cover (Adam suggested 8 items)
  • What a standard monthly meeting should hit (10 items)
  • What a standard quarterly review covers
  • How DOCFs should present data consistently across clients

Every DOCF currently runs meetings their own way, uses their own spreadsheets, and presents data in their own format. ImpactOS, paired with the meeting system (PRD-18), is the forcing function to standardize this.

Product needs

The product needs to support not just "view your metrics" but "run your meeting." This means:

  • Dashboard as a presentation tool during client meetings
  • Standard agenda templates that map to dashboard sections
  • Meeting prep that pulls the right metrics and documents together automatically
  • Consistency across DOCFs — if one client gets a professional review and another gets ad-hoc spreadsheets, the product isn't doing its job

New team member onboarding

Matthew (new DOCF, week 3) comes from a competitor firm and from school leadership. Even someone who "knows this world" is struggling to understand how CI operates because there's no documented standard. ImpactOS could serve as the onboarding tool — "this is how we present data, this is the standard meeting flow, this is what you show clients" — but only if the operational standards are defined first.

Section 09

Content Sign-Off Scope

A clarification was reached on what "content sign-off" means for pilot readiness.

In Sign-Off Scope

DOCFs validate accuracy of current data from systems of record

  • Is the enrollment number correct as of the reference date?
  • Is the GL data flowing correctly into financial metrics?
  • Are the calculations applying the right formulas to the right data?

Not in Sign-Off Scope

Design and operational decisions handled separately

  • Whether actuals match the forecast file (design/operational decision, not data fidelity)
  • Whether the DOCF prefers forecast numbers over actuals (view mode problem)

This distinction separates two types of overrides: "the data is wrong" (sign-off scope) vs. "I want different data" (product design scope).

Section 12

Summary: What Exists, What's Missing, What's Blocked

Exists and Working

  • Sage Intacct integration pulling financial actuals
  • SIS integration pulling enrollment, ADA, demographics
  • Dashboard with card-based KPIs, threshold status, trend indicators
  • Override system (enter value, attribution, rationale, audit trail)
  • Meeting system (PRD-18) for structured client meetings
  • Document sharing
  • View mode filtering logic in code (metricValues.ts)
  • Multi-school/org structure
  • DOCF management console

Designed but Not Yet Functional

  • Actuals/Forecast/Blended toggle (PRD-03 specifies it; code has filtering logic; no forecast data exists)
  • Forecast file upload/import mechanism
  • Per-metric methodology notes visible to EDs
  • Override reporting for admin governance (#167 in progress)
  • Restricted net assets / grant spend-down metric (in backlog, blocked by balance data)
  • Historical trend data for enrollment/ADA (financial history available; SIS history may need separate work)

Missing and Not Yet Designed

  • Side-by-side actuals vs. forecast view (requested by DOCFs; alternative to toggle)
  • Per-metric view mode (actuals for enrollment, blended for revenue — Phase 2 in PRD-03)
  • Budget variance tables (the legacy Domo-style view — actuals vs. budget by GL category)
  • Standard meeting agenda templates connected to dashboard sections
  • Working budget vs. approved budget distinction
  • Progressive override restriction (ability to disable overrides per metric over time)

Blocked by Operational Decisions

  • Metric definition standards — which number is "right" for each metric
  • Forecast file disposition — does it move into Sage or get ingested separately?
  • DOCF operating standards — consistency in how data is presented to clients
  • Override policy — what constitutes a legitimate override vs. a view mode preference
